# Impeccable — SETI Design Context

This file is the single source of truth for design decisions in this project. All Impeccable skills (`craft`, `layout`, `typeset`, `colorize`, `polish`, etc.) MUST read this before any design work.

---

## Design Context

### Users

Tabletop gamers who already know SETI: Search for Extraterrestrial Intelligence from the physical board game. They understand the game's rules, factions, and rhythms — they are here to play, not to learn. They expect the digital version to *preserve the ritual* of the tabletop experience (card geometry, faction colors, resource economy) while replacing only the manual overhead (bookkeeping, setup, state tracking).

Context of use: focused sessions at a computer, dark room, multi-player online. Not mobile-first, not a quick-check dashboard tool.

Job to be done: play a game of SETI with other people, end-to-end, without friction, and feel like they're stepping into a real mission control console.

### Brand Personality

Three anchor words:

- **Cutting-edge** — current-decade sci-fi, not retro CRT nostalgia, not Star Trek pulp. Precision hardware, not fantasy.
- **Scientific** — instrumentation, readouts, coordinates, frequency dials. Every number feels *measured*, not decorative.
- **Dark as deep space** — the void is the canvas. Interfaces emerge from darkness; they don't paper over it.

Tone in copy: economical, factual, faintly mission-log. Avoid gamification cheer ("Great job!"), avoid marketing fluff, avoid cute.

Emotional goal: calm concentration punctuated by moments of quiet awe when a signal is detected.

### Aesthetic Direction

**High-level metaphor**: a modern radio-astronomy observation deck aboard a research spacecraft. Ambient darkness with focused pools of instrument light. The interface is the glass between the player and the void.

**Theme**: dark, always. No light mode. Darkness is a brand pillar, not a preference.

**Color approach** (OKLCH throughout):

- Base void: `oklch(0.11 0.02 260)` — a deep indigo-black tinted toward blue-violet. Never pure `#000`.
- Surface layers: `oklch(0.17 0.025 260)`, `oklch(0.23 0.03 260)` — depth through lightness steps, NOT through box-shadows.
- Borders: `oklch(0.32 0.04 260)` at most; prefer hairline `oklch(0.28 0.035 260)`.
- Text primary: `oklch(0.96 0.008 260)` — off-white, faintly tinted toward the base hue for cohesion.
- Text dim: `oklch(0.72 0.015 260)`. Text faint: `oklch(0.52 0.02 260)`.
- **Primary accent — anodized blue**: `oklch(0.68 0.11 240)`. Used for player agency (active turn, primary CTAs, new-signal events, focused interactive elements). Muted, cool, clearly *metal* rather than *neon*. This is the one chrome accent.
- **Button / metal gradient range** (for the metal-face technique, see §Metal Surfaces below):
  - Highlight edge: `oklch(0.78 0.07 240)` (light catching the top bevel)
  - Face top: `oklch(0.58 0.10 240)`
  - Face bottom: `oklch(0.44 0.08 240)`
  - Shadow edge: `oklch(0.28 0.06 240)` (dark lower bevel)
- **Data readouts**: plain near-white `oklch(0.96 0.008 260)` in `JetBrains Mono`. No secondary color. Data is honest instrumentation — white phosphor on dark.
- Danger: `oklch(0.65 0.16 28)`. Reserved for destructive / game-ending actions only.

**Alien-faction palette (locked game data, reserved exclusively for game entities — never chrome):**
Centaurians, Exertians, Mascamites, Oumuamua, Amoeba, Glyphids, Anomalies — use the existing hex values from `tailwind.config.ts`. Treat them as *content color* (like a map legend), not *interface color*. Chrome stays amber/ice/neutral regardless of which faction is in focus.

**Typography** (none from the Impeccable reflex-reject list):

- **Display**: `Unbounded` (Google Fonts, variable 200–900). Wide, geometric, slightly futuristic without being trope-y. Use at large sizes for screen titles, phase names, end-of-game reveals.
- **Body**: `Hanken Grotesk` (Google Fonts, variable). Neutral, engineered, reads at any size. Replace the current `Ubuntu`.
- **Mono / readouts**: `JetBrains Mono` (Google Fonts). For coordinates, frequencies, tech values, resource counts, timestamps, log entries. NOT for UI chrome or body copy.

Type scale: fixed `rem` scale with a 1.25 ratio minimum between steps (this is a game UI, not a marketing page — no `clamp()` on product chrome).

**Motion**: restrained. One well-orchestrated moment of reveal (e.g. signal detection, turn transition, end-of-round) is worth more than ten hover twitches. Exponential ease-out. No bounces. Honor `prefers-reduced-motion` by default for safety (cheap to implement).

**Decorative vocabulary** — purposeful, not decorative:

- Subtle star/noise field layered into the base background (low-opacity, static — not animated).
- Signal-waveform divider (a single thin line with a faint amplitude) as an occasional section break.
- Instrument-style tick marks for sliders, timers, and phase progress.
- Do NOT: use HUD corner accents on every panel, do NOT use grid overlays everywhere, do NOT use glassmorphism as chrome.

### Metal Surfaces

The defining tactile quality of the interface. Borders and interactive surfaces should feel like anodized/brushed metal panels lit from above, not flat CSS rectangles. This is deliberate, controlled skeuomorphism — restrained enough to read as *engineered*, never crossing into Aqua/Lion kitsch.

**Hairline metal border** (default for panels, inputs, dividers — the workhorse):

```css
/* single utility; reach for this before any other border */
border: 1px solid transparent;
border-image: linear-gradient(
  to bottom,
  oklch(0.55 0.04 240) 0%,
  oklch(0.32 0.03 240) 50%,
  oklch(0.22 0.025 240) 100%
) 1;
/* paired with an inset highlight to catch the top edge */
box-shadow: inset 0 1px 0 oklch(0.78 0.04 240 / 0.25);
```

Alternative for rounded corners (where `border-image` breaks): use a 1px solid border at `oklch(0.34 0.035 240)` plus a layered `box-shadow` with both an inset highlight (top) and inset shadow (bottom).

**Metal button face** (primary CTAs, active-turn indicators, any surface that demands touch):

```css
background: linear-gradient(
  to bottom,
  oklch(0.58 0.10 240) 0%,
  oklch(0.50 0.09 240) 50%,
  oklch(0.44 0.08 240) 100%
);
border: 1px solid oklch(0.30 0.06 240);
box-shadow:
  inset 0 1px 0 oklch(0.78 0.07 240 / 0.5),     /* top highlight — the light catch */
  inset 0 -1px 0 oklch(0.22 0.05 240 / 0.6),    /* bottom bevel shadow */
  0 1px 0 oklch(0.1 0.02 260 / 0.8),             /* contact shadow */
  0 0 0 1px oklch(0.2 0.04 240 / 0.4);           /* outer hairline */
color: oklch(0.98 0.01 240);
/* hover: brighten gradient 4–6% in lightness, unchanged chroma */
/* active (pressed): invert — darker at top, lighter at bottom, drop the contact shadow */
```

**Secondary / ghost button**: no fill, just the hairline metal border + soft inset top-highlight. Hover fills with `oklch(0.22 0.03 260)` at 40% opacity.

**Metal rules**:

- Metal borders are always **cooler than** the surface they sit on. Warm metal = brass = wrong metaphor.
- Never combine a metal gradient with a glow. Metal reflects; it doesn't emit. Glow belongs to signal events (detection, win state), not chrome.
- Keep metal confined to: borders (hairline), primary buttons, active-state indicators, slider tracks/handles, toggle switches. Do NOT metallize every surface — overuse kills the effect immediately.
- A panel background is NOT metal. Panels are dark surface; the metal is only their frame.

**Anti-references — explicitly not this:**

- Not a **SaaS dashboard**. No metric tiles with a big number + tiny label + sparkline. No "At a glance" cards. No Linear/Vercel-admin aesthetic.
- Not **Mars colonization** / Terraforming Mars rust-on-dark. The existing `brand-guidelines` skill describes a different project; ignore its Mars palette for this codebase.
- Not **retro sci-fi** (no CRT scanlines, no 80s neon, no Star Wars wipes).
- Not **cute / gamified** (no mascots, no confetti on actions, no rounded-pastel everything).
- Not **generic dark mode with glowing purple-to-cyan gradients**. That is the #1 AI-slop tell.

### Design Principles

1. **Signal over noise.** The screen is dark by default. Only the information relevant *right now* is luminous. Accents are rare; their power comes from scarcity, not saturation.
2. **Instrumentation, not decoration.** Numbers, coordinates, counts, and statuses are rendered like lab instruments: mono, aligned, high-contrast, precise. They are never dressed up as marketing stat tiles.
3. **Depth through darkness and metal.** Layers are built from two techniques only: stepped OKLCH lightness for surfaces, and hairline metal borders for edges. No drop-shadows for hierarchy; no glow for chrome.
4. **One accent, one role.** Anodized blue is the *only* chrome accent — player agency, CTAs, focused interactive elements. Data readouts are white mono, not a second color. Alien-faction colors stay strictly on game entities.
5. **Card geometry is sacred.** Game-card dimensions, aspect ratios, visual styles, and faction colors are immutable. Design *around* them. Never restyle them to fit a layout — restructure the layout instead.

## Implementation Constraints

- **Only target**: `packages/client` (Vite + React 18 + Tailwind v4). `packages/web` (Next.js) is out of scope for new design work.
- **Accessibility bar**: keep it simple. No formal WCAG target, no colorblind-safe remapping of faction colors. Don't over-engineer — but don't regress below sensible defaults (keyboard focus visible, form labels present, reasonable contrast on text).
- **Existing dependencies to use**: `@radix-ui/react-select`, `class-variance-authority`, `clsx`, `tailwind-merge`, `zustand`, `@tanstack/react-router`, `@tanstack/react-query`. Do not introduce new UI libraries without reason.
- **Sacred files — do not restyle**:
  - `packages/client/src/styles/globals.css` — the `.seti-sector-grid` and `.cgo-*` / `.card-*` classes are game-card rendering. Do not touch.
  - Any card component that renders game cards directly.
- **Fonts**: load `Unbounded`, `Hanken Grotesk`, and `JetBrains Mono` from Google Fonts. Remove the current `Ubuntu` import from `globals.css` once all usages are migrated.
- **Color tokens**: migrate `packages/client/src/config/theme.ts` to OKLCH values described above. Existing consumers reading `THEME_TOKENS` should keep working (keep the shape of the object; change the values).
