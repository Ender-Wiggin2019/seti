## Critical Issues

> Issues that break correctness, lose data, or violate game rules. Must fix.

### C-01: Planetary moon model collapses multi-moon planets into a single slot

**File:** `packages/server/src/engine/board/PlanetaryBoard.ts`
**Line(s):** 52-58, 150-161, 224-227, 278-284
**Dimension:** Logic

`PlanetaryBoard` currently stores moon occupancy as a single `moonOccupant`, and moon landing is blocked once that single field is filled. This contradicts board config where planets like Jupiter and Saturn have multiple moon slots. In solo mode this also breaks Rival moon landing priority behavior and invalidates left/right moon-choice rules on cards.

**Current:**
```ts
moonOccupant: IMoonOccupant | null;
...
if (isMoon) {
  planetState.moonOccupant = { playerId };
}
...
if (planetState.moonOccupant !== null) {
  return false;
}
```

**Suggested fix:**
Replace single `moonOccupant` with a moon-slot array (size from `moonSlots` or `landingSlotKinds`), then make landing choose a specific moon slot (including decision-direction tie-break for rival).

---

### C-02: Rival orbit/land reward resolver silently drops `signal` rewards

**File:** `packages/server/src/engine/solo/RivalActionResolver.ts`
**Line(s):** 574-595
**Dimension:** Logic

`applyPlanetRewards` handles `resource/card/tuck/exofossil` but ignores `signal` rewards from planet mission config (e.g. Mercury/Mars orbit rewards). For rival probe placement actions, this causes legal planet rewards to be lost, changing progress, data, and sector outcomes.

**Current:**
```ts
switch (reward.type) {
  case 'resource': ...
  case 'card': ...
  case 'tuck': ...
  case 'exofossil': ...
  default:
    break;
}
```

**Suggested fix:**
Add `signal` handling in rival planet reward resolution (target planet sector marking with existing signal-placement priority rules), and add dedicated tests for orbit rewards containing signal entries.

## Warnings

> Issues that may cause bugs, hurt maintainability, or deviate from architecture. Should fix.

### W-01: Solo tech-order config is duplicated in server instead of shared config

**File:** `packages/server/src/engine/solo/RivalActionResolver.ts`
**Line(s):** 65-73
**Dimension:** Maintainability / Configurability

`RIVAL_TECH_CATEGORY_ORDER_BY_BOARD` is hardcoded in server logic while solo board config is already data-driven in `@seti/common/constant/solo`. This increases drift risk between server decision logic and shared solo metadata.

**Suggested improvement:** Move this mapping into `common/constant/solo.ts` and consume it from server.

---

### W-02: Rival action resolver is oversized and mixes many domains

**File:** `packages/server/src/engine/solo/RivalActionResolver.ts`
**Line(s):** 1-1285
**Dimension:** Maintainability

Single class handles probe pathing, signal priority, tech selection, five alien special flows, and trace rules. This file size and coupling make regressions more likely when adding new solo cards/species behavior.

**Suggested improvement:** Split by domain (`RivalProbeResolver`, `RivalTelescopeResolver`, `RivalAlienSpecialResolver`) as already planned in architecture doc.
