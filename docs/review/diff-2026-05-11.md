## Critical Issues

### C-01: Lockfile was downgraded and contains unrelated dependency churn

**File:** `pnpm-lock.yaml:1`, `package.json:5`
**Dimension:** Quality / Maintainability

`package.json` declares `packageManager: "pnpm@10.0.0"`, but the uncommitted lockfile changes `lockfileVersion` from `9.0` to `6.0` and rewrites a large dependency graph. The manifest changes only add scripts, so this lockfile churn is not explained by dependency changes. A frozen install under the declared package manager is likely to fail or immediately rewrite the lockfile again.

**Suggested fix:** regenerate or restore `pnpm-lock.yaml` with the declared pnpm version and keep only lockfile changes that are directly required by manifest dependency edits. For this diff, the lockfile likely should not change.

## Warnings

### W-01: Image-mode tech overlays ignore board-specific tech order

**File:** `packages/client/src/features/solo/RivalPanel.tsx:56`, `packages/client/src/features/solo/rivalBoardPresentation.ts:46`
**Dimension:** Logic / Client Architecture

`TECH_CATEGORY_ORDER` and `RIVAL_BOARD_TECH_POINTS` are fixed as Probe / Scan / Computer for all boards. The solo server and docs define board-specific order: board 2 is Scan / Probe / Computer, board 3 is Computer / Scan / Probe, and board 4 is Probe / Computer / Scan. For example, `rival-board-3` will render the Probe tech tile where the board expects Computer.

**Suggested fix:** move the board-specific category order/positions to a shared common constant or mirror the existing server mapping, pass `boardConfigId` into the overlay renderer, and add tests for at least `rival-board-2`, `rival-board-3`, and `rival-board-4`.

### W-02: Image mode no longer renders Rival data pool

**File:** `packages/client/src/features/solo/RivalPanel.tsx:235`, `packages/client/src/features/solo/RivalPanel.tsx:482`
**Dimension:** Completion / Client Architecture

Text mode still renders `rival.computer.dataPool` as `Pool {{count}}`, but image mode only renders `filledSlots`. `IPublicRivalComputerState.dataPool` is public Rival state and the solo architecture calls it out as separate state that must remain visible. This is a user-visible data loss in the new image-mode board.

**Suggested fix:** add a compact data-pool readout overlay in `RivalImageModeBoard` and extend `RivalPanel.test.tsx` / E2E image-mode assertions to check the count.

### W-03: `TechBonusEffect.apply(CARD)` still implements the old deck-only behavior

**File:** `packages/server/src/engine/effects/tech/TechBonusEffect.ts:91`
**Dimension:** Logic / Maintainability

The new tech flow correctly routes printed `CARD` bonuses through `createAnyCardChoice`, but the public `TechBonusEffect.apply()` method still draws from the deck for `ETechBonusType.CARD`, and its unit test still asserts that old behavior. A future caller can reintroduce the original bug by using the generic apply path.

**Suggested fix:** remove `CARD` from the non-interactive `apply()` path, or make it return `applied: false` / throw with a clear message directing callers to `createAnyCardChoice`. Update the unit test so the public contract matches the new rule semantics.

### W-04: New singleton script fails Biome checks

**File:** `scripts/check-react-singleton.mjs:2`, `scripts/check-react-singleton.mjs:67`, `scripts/check-react-singleton.mjs:71`
**Dimension:** Code Quality

**Suggested fix:** apply Biome import ordering and either configure an exception for this CLI script or use the project-approved output pattern for command-line scripts.

### W-05: Current staged set is incomplete for a working commit
