# Code Review: 1v1 flow mock/debug coupling

**Date:** 2026-05-05
**Scope:** Lobby create/join/start flow, game websocket join/action flow, and debug/local game paths related to two-player play.
**Reviewer:** AI Code Review Skill

## Summary

| Severity | Count |
|----------|-------|
| Critical | 0 |
| Warning  | 3 |
| Suggestion | 1 |

**Overall assessment:** The normal `/lobby -> /room/:id -> /game/:id` 1v1 path is not using the client-side `player-1` / `player-2` local mock state. It creates a DB room, inserts real authenticated users into `gamePlayers`, starts a real `Game` with those user ids, and websocket actions are keyed by the JWT user id. The main concern is not direct mock leakage in the normal client path; it is that debug/scenario behavior is implemented inside production-facing lobby/gateway modules, which makes the natural business flow easier to bypass and harder to reason about.

## Warnings

### W-01: Public room creation accepts scenario presets that mutate game state at start

**File:** `packages/server/src/lobby/lobby.controller.ts`, `packages/server/src/lobby/lobby.service.ts`
**Line(s):** `lobby.controller.ts:38-48`, `lobby.service.ts:19-26`, `lobby.service.ts:70-72`, `lobby.service.ts:264-281`
**Dimension:** Maintainability / Flow integrity
**Status:** Fixed on 2026-05-05.

The normal lobby controller accepts `scenarioPreset` from the request body and stores it in room options. `startGame()` then applies test/debug scenario mutators such as `applyBehaviorFlowScenario`, `applySpendSignalTokenScenario`, and `applyDeliverSampleScenario`.

This does not look like a 1v1 mock bug in the UI, because `CreateRoomDialog` does not submit `scenarioPreset`. But the API still exposes a path where a regular authenticated lobby room can start from a pre-mutated state, bypassing the natural setup/start progression.

**Suggested fix:**
- Move scenario preset handling out of `LobbyService` into debug-only endpoints, or gate it behind an explicit non-production/debug flag.
- Keep public lobby room options limited to real business settings: player count, enabled aliens, undo, timer, expansion flags.

**Resolution:**
- Removed `scenarioPreset` forwarding from `LobbyController.createRoom()`.
- Removed `scenarioPreset` storage and scenario mutator execution from `LobbyService`.
- Added regression coverage that public lobby payloads cannot persist scenario presets.

### W-02: Debug bot automation is embedded in the production game gateway

**File:** `packages/server/src/gateway/game.gateway.ts`
**Line(s):** `game.gateway.ts:28`, `game.gateway.ts:56-63`, `game.gateway.ts:145`, `game.gateway.ts:177`, `game.gateway.ts:194`, `game.gateway.ts:212`, `game.gateway.ts:230`, `game.gateway.ts:337-530`
**Dimension:** Maintainability / Flow integrity

The gateway calls `runDebugBots()` after join/action/end-turn/free-action/input events. The method is guarded by `DebugSessionRegistry.isDebugGame()`, so ordinary DB-created lobby games should not auto-play. However, the gateway still owns bot action selection, random input response generation, and random movement.

That makes the main realtime path harder to audit because real event delivery and debug automation are interleaved in one class.

**Suggested fix:**
- Extract debug bot automation behind a debug service/module boundary.
- Have gateway publish real event results only; debug endpoints can explicitly invoke bot advancement for registered debug sessions.

### W-03: Debug sessions create valid JWTs for generated users and can persist games

**File:** `packages/server/src/debug/debug.service.ts`
**Line(s):** `debug.service.ts:116-153`, `debug.service.ts:174-201`, `debug.service.ts:217-250`, `debug.service.ts:283-322`
**Dimension:** Flow integrity

Debug sessions generate random players, sign JWTs for them, register bot users, and sometimes persist the game through `GameRepository.create()`. This is appropriate for replay tooling, but it means debug games can look very similar to real games at the gateway/auth layer.

The risk is operational rather than a direct 1v1 UI bug: if debug endpoints are exposed in an environment where they should not be, they can create artificial game sessions that bypass lobby create/join/start.

**Suggested fix:**
- Ensure all debug controllers are disabled or protected outside local/test environments.
- Add a clear persisted marker for debug games if they remain persisted, so listing/analytics/admin tools can exclude them.

## Suggestion

### S-01: Keep local mock page isolated from business navigation

**File:** `packages/client/src/pages/game/GameDebugPage.tsx`, `packages/client/src/routes.tsx`
**Line(s):** `GameDebugPage.tsx:327-349`, `GameDebugPage.tsx:756-803`, `GameDebugPage.tsx:1085-1090`, `GameDebugPage.tsx:1202-1209`, `GameDebugPage.tsx:1327-1337`, `GameDebugPage.tsx:1581-1584`, `routes.tsx:97-125`
**Dimension:** Maintainability
**Status:** Fixed on 2026-05-05.

The `player-1` / `player-2` hardcoding is concentrated in `/debug/game` local mode. That is acceptable as a visual/debug fixture as long as it stays out of the normal app shell and lobby/game path.

**Suggested fix:**
- Keep `/debug/*` routes excluded from production builds or protected behind a debug flag.
- Avoid linking debug pages from ordinary lobby/room/game UI.

**Resolution:**
- Added `VITE_ENABLE_DEBUG_ROUTES`, defaulting to disabled.
- Wrapped `/debug/game`, `/debug/replay`, `/debug/solar`, and `/debug/server` routes so disabled environments redirect to `/`.
- Added client env regression coverage for the disabled default.

## Positive Observations

- `LobbyService.startGame()` builds player identities from `gamePlayers` joined to `users`, not from fixed ids.
- `GameGateway` derives the acting player from the socket JWT via `getUserId(client)`, not from client-supplied `playerId`.
- `GameContextProvider` uses the authenticated store user id as `myPlayerId`; the normal game page does not force `player-1`.
- `CreateRoomDialog` submits only normal game options and does not expose scenario presets.
