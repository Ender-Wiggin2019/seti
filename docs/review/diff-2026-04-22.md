## 2. 实现完整性（client & server）

### Server 层

- ✅ `Game.assertCanTakeTurnAction` 正确地把守卫放在 phase/turn 检查之前，错误码 `INVALID_INPUT_RESPONSE` 与现有 pending-input 分支一致。
- ✅ `restoreSetupTuckPromptsIfNeeded` 在 `getGame`（DB 冷加载）时调用，且缓存命中路径不会重复执行，避免重复 rebuild。
- ⚠️ **潜在缺陷**：守卫与 restore 都用 `tuckedIncomeCards.length === 0 / > 0` 作为"是否完成 setup tuck"的判据。当前 `BaseCorporation.startActions.tuckIncome = 1` 下 OK；**一旦出现 `tuckIncome > 1` 的企业卡，处于 "已 tuck 1 / 需 tuck 2" 中间态的玩家会被判定为"已完成"**，restore 不会重建链，`assertCanTakeTurnAction` 也不再阻塞。
- ⚠️ **`onComplete` 链路丢失**：`GameSetup.createSetupTuckInputChain` 是递归的，而 `restoreSetupTuckPromptsIfNeeded` 调用的是无 `onComplete` 的 `TuckCardForIncomeEffect.execute(player, game)`。冷加载后若企业需要 2 次 tuck，仅剩 1 次。这两点都指向同一个抽象泄漏：**"setup tuck 语义"被复制在了 `GameSetup` 和 `GameManager` 两处，且二者可能分叉**。
- ⚠️ **`undo` 路径未覆盖**：`GameDeserializer` 中 `concretePlayer.waitingFor = undefined`。undo 流程通过 `deserializeGame(checkpoint.stateDto)` 也会抹掉 `waitingFor`；只不过 `captureCurrentTurnCheckpoint` 在 `turnIndex <= 0` 时跳过，setup 阶段一般没有 checkpoint，实战中未必踩到，但**语义上 restore 应与 deserialize 同层**，放在 `GameDeserializer` 末端（或作为 `deserializeGame` 的 post-hook）更自洽。

### Client 层

- ❌ **无对应 UI 反馈**：当玩家 A 已完成 setup tuck、玩家 B 还未完成时，A 的客户端并不知道"不能 PASS"。服务端抛 `INVALID_INPUT_RESPONSE` 后，UI 只能以 toast 形式兜底。`IPublicGameState` 没有广播"有其他玩家仍在 setup-tuck"这个信号，因此 A 的 PASS 按钮无法在客户端预先禁用。
- ❌ **`HandView` 和 `card.css` 改动与主题无关**：一个 e2e 修复的 commit 顺带带入了 dock 缩放和 line-height 调整，**违反单一目的 commit 原则**（`CLAUDE.md` 第 3 条"同步修改不要不一致"隐含的可追溯性也会受影响）。建议 rebase 拆分。
- ⚠️ **`scale-[0.52] + origin-top-left` 会造成"视觉缩小但 layout 盒子不变"**：外层 `button` 仍按原 `CardRender` 尺寸占位，右下会留空白区域，并且 hover `-translate-y-0.5` 的视觉中心与实际可点击区域不匹配。若意图是缩放整个卡片，应在 `button` 上给定固定 `width/height` 或使用 `transform-gpu` + 外层 flex 占位；否则建议用 CSS zoom 或 container query 响应式字号。

### 整体完整性评分

服务端逻辑在**当前配置下**工作正确，但对扩展（多次 setup tuck、undo 回滚、client UX）的前瞻不足。**评分：6.5/10**。

private restoreSetupTuckPromptsIfNeeded(game: IGame): void {
  if (game.round !== 1) return;
  for (const player of game.players) {
    if (player.waitingFor) continue;
    if (player.tuckedIncomeCards.length > 0) continue;
    const setupInput = TuckCardForIncomeEffect.execute(player, game);
    if (setupInput) player.waitingFor = setupInput;
  }
}

**建议**：在 `GameSetup` 中暴露一个纯静态方法 `GameSetup.buildSetupTuckInputFor(player, game): IPlayerInput | undefined`，既用于初始化，也用于 rehydrate。`assertCanTakeTurnAction` 则改为检查 `player.hasCompletedSetupTuck`（在 `Player` 上持一个计数或布尔），语义明确且不依赖 `tuckedIncomeCards.length` 这种耦合内部表达的启发式。

#### 3.2 rehydrate 位置不对

`GameDeserializer` 主动把 `waitingFor = undefined`（丢弃），而 `GameManager.getGame` 再事后补回——**写和读不对称**。建议二选一：

- 方案 A（推荐）：在 `GameDeserializer` 中遍历玩家，若 `round === 1` 且 tuck 未完成则调用 `GameSetup.buildSetupTuckInputFor` 重建；这样 `deserializeGame` 产出的 `IGame` 是"可直接使用的"，`GameManager` 无需知道 setup 语义。
- 方案 B：干脆持久化 `waitingFor` 的 DTO（至少 setup-tuck 这类确定性输入），让 `GameSerializer.waitingFor: null` 消失。

#### 3.3 `hasUnresolvedSetupTuck` 逻辑冗余

```ts
const hasSetupPrompt = this.players.some((player) => player.waitingFor);
const hasUnresolvedSetupTuck =
  this.round === 1 &&
  hasSetupPrompt &&
  this.players.some(
    (player) =>
      player.tuckedIncomeCards.length === 0 && player.hand.length > 0,
  );
```

- `hand.length > 0` 的必要性未说明，且 setup 阶段玩家必然有手牌，属于无用条件（或需注释为什么）。
- `hasSetupPrompt` 与 `tuckedIncomeCards.length === 0` 互为冗余（若有玩家未 tuck 且 `waitingFor` 已被 `restoreSetupTuckPromptsIfNeeded` 补上，则一定 `hasSetupPrompt`；若 restore 没跑，则表示在内存路径内还没触发过，此时 `waitingFor` 也一定仍在）。化简为：

```ts
const hasUnresolvedSetupTuck =
  this.round === 1 &&
  this.players.some((player) => !player.hasCompletedSetupTuck);
```
