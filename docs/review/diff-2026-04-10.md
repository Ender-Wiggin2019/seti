## Critical Issues

### C-01: Completed mission state is now leaked to every client

**File:** `packages/server/src/persistence/serializer/GameSerializer.ts:426-434`
**Line(s):** 426-434
**Dimension:** Logic / Quality

`toPublicPlayerState` now copies `player.completedMissions` and `player.endGameCards` for all viewers without an `isViewer` guard, even though the PRD explicitly marks completed missions as face-down private information. The shared type still declares `completedMissions?: string[]`, but the serializer pushes full `TCardItem` payloads, so other players gain perfect knowledge of your hidden scoring cards and the runtime shape no longer matches `IPublicPlayerState`. This violates the rule spec (docs/arch/prd-rule.md:117-125) and breaks fog-of-war mechanics.

**Current:**
```ts
playedMissions: cloneValue(player.playedMissions as never),
completedMissions: cloneValue(player.completedMissions) as string[],
endGameCards: cloneValue(player.endGameCards as never),
```

**Suggested fix:** Only project completed/end-game mission detail when `isViewer` is true; otherwise send counts or `undefined`. Keep the contract honest by changing the type to reflect the payload that actually ships.

## Warnings

### W-01: Debug-only `DUMMY` alien now joins the live selection pool

**File:** `packages/common/src/types/BaseCard.ts:75-99`
**Line(s):** 75-99
**Dimension:** Logic / Configurability

Adding `EAlienType.DUMMY` and immediately registering `DummyAlienPlugin` causes `GameSetup` to shuffle over an eight-entry enum (see `packages/server/src/engine/GameSetup.ts:43-48`), so the placeholder species can be randomly chosen for hidden aliens in real games. The PRD limits base mode to “2 selected from 5” aliens (docs/arch/prd-rule.md:23-28); DUMMY has no art, slots, or balancing, so surfacing it breaks rule fidelity. If it is meant for dev/testing, it needs to be opt-in (e.g., behind a feature flag or removed from `Object.values(EAlienType)`), not part of the production pool.

**Suggested fix:** Keep DUMMY out of `ALL_ALIENS` / selection unless a debug flag is enabled, or move the placeholder into a separate enum used only by tests.

### W-02: `markTrace` now crashes whenever the stub game lacks `alienState`

**File:** `packages/server/src/engine/cards/BehaviorExecutor.ts:320-333`
**Line(s):** 320-333
**Dimension:** Maintainability / Quality

`buildMarkTraceAction` now assumes `game.alienState` is always defined and calls `createTraceInput` unconditionally. The change immediately breaks `BaseRepresentativeCards` (see `packages/server/__tests__/engine/cards/BaseRepresentativeCards.test.ts:120-136`), and any other BehaviorExecutor consumer that uses lean `game` doubles (e.g., mission previews, offline card tests) now explodes with “Cannot read properties of undefined (reading 'createTraceInput')”. At minimum you need a guard or a simple fake alien state for the executor so tests and tooling can still run.

**Suggested fix:** Either (a) keep the previous counter increment when `game.alienState` is missing, or (b) update all test helpers/fixtures to construct an `AlienState` stub before executing behaviors. The current state blocks the server test suite.

### W-03: Custom DESC tokens/tests no longer align and unhandled effects silently disappear

**Files:** `packages/server/src/engine/cards/Behavior.ts:92-99`, `packages/server/src/engine/cards/BehaviorExecutor.ts:347-360`, `packages/server/__tests__/engine/cards/Behavior.test.ts:38-44`
**Dimension:** Maintainability / Quality

Switching `appendCustom` to prefer `effect.desc` was necessary for deterministic handler keys, but the surrounding surface was not updated: `Behavior.test` still expects `behavior.custom` to contain the generated UUID, so vitest now fails. Simultaneously, unhandled custom IDs used to emit `CARD_CUSTOM_EFFECT_UNHANDLED` events; now they are silently dropped (`return undefined`), which removes the only runtime signal that a DESC card still lacks a handler. Together this regresses both test coverage and production observability.

**Suggested fix:** Update the tests (and any downstream consumers) to assert against the new deterministic key *and* reintroduce some form of warning/log when a custom handler is missing. That preserves the benefits of stable tokens without losing regression detection.

### W-04: Milestone UI hides remaining capacity after the first claim

**File:** `packages/client/src/pages/game/GameLayout.tsx:403-455`
**Line(s):** 403-455
**Dimension:** Logic / Maintainability

`buildMilestoneItems` only pushes an “open” entry when `resolvedPlayerIds.length === 0`, so as soon as one player claims the 25/50/70 VP milestone (or one of the duplicated neutral thresholds), the UI shows the bucket as fully claimed even though more slots remain. Neutral milestones are modeled with `markersRemaining` (see `packages/server/src/engine/scoring/Milestone.ts:14-59`) precisely because 2p/3p games carry multiple neutral tokens per threshold. Players can no longer tell whether a milestone is still contestable, which conflicts with section 10.2 of the PRD.

**Suggested fix:** Emit “open” entries for the remaining capacity—e.g., add one placeholder per unfilled gold slot and per `markersRemaining` neutral slot—so the UI reflects the actual milestone state.
