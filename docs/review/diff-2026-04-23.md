## Warnings

### W-01: `createRoomByUi` now bypasses the real post-create navigation

**File:** `packages/e2e/helpers/real-flow.ts`
**Line(s):** 149-152
**Dimension:** Quality

After clicking the real submit button, the helper reads the response body and performs `page.goto(`/room/${id}`)` itself. That means every E2E case using `createRoomByUi()` stops verifying whether the actual client flow navigates to the room page, closes the dialog, and hydrates the room state correctly. A broken router transition or missing redirect would now stay green in CI.

**Suggested fix:** Wait for the URL/UI state produced by the real client flow after submit, and only use the response body for assertion or fallback diagnostics.

---

### W-02: PASS sync helper can succeed without any log propagation

**File:** `packages/e2e/helpers/real-flow.ts`
**Line(s):** 326-330
**Dimension:** Quality

`clickPassAndWaitForLogSync()` now returns as soon as turn ownership flips, even when neither page's event log count increases. Because the updated tests mostly call `openEventLog()` after this without asserting a new entry, the suite no longer proves that the PASS event replicated to both clients' logs. This weakens the original synchronization guarantee and can hide regressions in event broadcasting.

**Suggested fix:** Keep log-count growth as a required condition for this helper, or split the helper into two explicit waits: one for turn handoff and one for event-log replication.

---

### W-03: Moon UI now marks every empty moon slot as `Open` for all players

**File:** `packages/client/src/features/board/PlanetCard.tsx`, `packages/client/src/features/board/PlanetaryBoardView.tsx`
**Line(s):** 133-140, 183-185
**Dimension:** Logic

The rules require moon landing permission from technology/effects, but the board/card UI now renders any empty moon slot as open/highlighted once the planet has moons. Since `moonUnlocked` was removed and no player-specific capability replaced it, the client now communicates that moons are generally available even when the viewer cannot legally land there. That is a rules/UX mismatch and will mislead players when evaluating legal land targets.

**Suggested fix:** Render moon slots as neutral informational markers, or pass viewer-specific moon-landing capability into the board/card components before labeling them as open.
