## Critical Issues

### C-01: Orbit/Land availability in the UI is computed from stale `planetSpaceId`, so a probe on a planet can still show no `orbit` / `land`

**File:** `packages/common/src/rules/planet.ts`
**Line(s):** 24-37
**Dimension:** Logic / Quality

`canOrbitPlanet()` still decides legality by comparing each probe's `spaceId` to `planet.planetSpaceId`. In the current server flow, that field is not maintained from the live solar-system state: it is only defined on `IPlanetState` / serialized as-is in `packages/server/src/persistence/serializer/GameSerializer.ts:467`, while the actual current planet positions are rebuilt in `packages/server/src/engine/utils/stateProjection.ts:30-64` as `solarSystem.planetSpaceIds`.

That means the UI path `ActionMenu -> getAvailableMainActions() -> canOrbit()/canLand() -> canOrbitPlanet()` can return `false` even when the probe is physically on the correct planet space. This matches the reported bug exactly.

**Current:**

```ts
export function canOrbitPlanet(
  planet: IPublicPlanetState,
  player: IPublicPlayerState,
  gameState: IPublicGameState,
): boolean {
  const planetSpaceId = planet.planetSpaceId;
  if (!planetSpaceId) {
    return false;
  }

  return gameState.solarSystem.probes.some(
    (probe) =>
      probe.playerId === player.playerId && probe.spaceId === planetSpaceId,
  );
}
```

**Suggested fix:**

Use the new shared solar-system helpers as the source of truth on the client side, for example resolve the current planet space from `gameState.solarSystem.planetSpaceIds` / `findPlanetSpaceId()` instead of trusting `planet.planetSpaceId`. Even better, remove this duplicate source of truth from `planetaryBoard` entirely once callers are migrated.

## Warnings

### W-01: Card-granted tech rotation is intentionally implemented against the written rule

**File:** `packages/server/src/engine/actions/ResearchTech.ts`
**Dimension:** Logic

Both `docs/arch/rule-simple.md:228-235` and `docs/arch/rule-raw.md:156-166` state that getting a tech from a card effect still rotates the solar system. The implementation now explicitly does the opposite: `packages/server/src/engine/actions/ResearchTech.ts:55-83` and `packages/server/src/engine/cards/BehaviorExecutor.ts:317-322` document a “decoupled” model where only a printed ROTATE icon causes rotation, and `packages/server/__tests__/engine/board/SolarSystem.test.ts:257-275` locks that behavior in.

If this is a deliberate product decision, the rule docs need to be updated. If the docs are the source of truth, the current behavior is a rule violation and should be corrected.

### W-02: Test coverage misses the exact cross-layer path that broke in production

**File:** `packages/common/__tests__/rules/actions.test.ts`
**Dimension:** Completion / Quality

The relevant tests are split by layer, but not connected:

- `packages/common/__tests__/rules/actions.test.ts:112-129` and `packages/common/__tests__/rules/planet.test.ts:67-79` use hand-built fixtures that always populate `planetSpaceId`.
- `packages/server/__tests__/persistence/serializer/GameSerializer.test.ts:38-50` checks that projection returns spaces, but it does not assert the new `planetSpaceIds` / `sectorSpaceIds` / `probeSpaceById` indexes or the compatibility between projected state and common action rules.
- `packages/client/src/features/actions/ActionMenu.tsx:294-296` consumes `getAvailableMainActions()` directly, but there is no test exercising that with a real `projectGameState()` result.

This is why the known UI bug survives while all touched tests still pass.
