## Critical Issues

### C-01: Oumuamua sector metadata becomes stale after solar-system rotation

**File:** `packages/server/src/engine/alien/plugins/OumuamuaAlienPlugin.ts`
**Line(s):** 153-157, 235-267
**Dimension:** Logic Correctness

`placeOumuamuaPlanet()` stores the discovery-time `sectorId` in a hidden alien-board slot, and `createSectorOrTileSignalInput()` later checks the requested sector against that stored value. The Oumuamua planet is on a rotating ring, so after any ring-3 rotation the planet's current sector is `game.solarSystem.getSectorIndexOfPlanet(EPlanet.OUMUAMUA)`, not the sector captured at discovery. This means scan/card effects will offer the Oumuamua tile choice in the old sector and fail to offer it in the current sector.

**Current:**

```ts
if (!meta || meta.sectorId !== sectorId) {
  const result = markSector();
  return onComplete?.(result);
}
```

**Suggested fix:** Keep `spaceId` as placement metadata if needed, but derive the current Oumuamua sector at decision time through a common helper, e.g. `getSectorByPlanet(game, EPlanet.OUMUAMUA)` or a `SolarSystem.getSectorIndexOfPlanet()` based helper. Add a regression test that discovers Oumuamua, rotates the solar system so ring 3 moves, then verifies only the new current sector offers `oumuamua-sector` / `oumuamua-tile`.

### C-02: The Oumuamua signal choice is not a bottom-level hook

**Files:** `packages/server/src/engine/effects/scan/MarkSectorSignalEffect.ts`, `packages/server/src/engine/cards/registerDescHandlers.ts`, `packages/server/src/engine/cards/base/AnySignalQuickMissionCard.ts`, `packages/server/src/engine/cards/base/HerschelSpaceObservatoryCard.ts`
**Line(s):** `MarkSectorSignalEffect.ts` 79-97, `registerDescHandlers.ts` 244, `AnySignalQuickMissionCard.ts` 155/165, `HerschelSpaceObservatoryCard.ts` 112/121
**Dimension:** Logic Correctness / Maintainability

The requested rule says when marking a sector that has Oumuamua, the player can choose the original sector or Oumuamua. The new `markByIndexWithAlternatives()` covers scan paths, but existing production callers still use `markByIndex()`, `markById()`, or `markOnSector()` directly, bypassing the Oumuamua branch entirely. Examples include `desc.et-14`, probe-sector quick mission marking, and Herschel sector marking.

**Current:**

```ts
MarkSectorSignalEffect.markByIndex(player, game, token.sectorIndex);
MarkSectorSignalEffect.markById(player, game, sectorIds[0]);
MarkSectorSignalEffect.markOnSector(player, game, sectors[0]);
```

**Suggested fix:** Put the Oumuamua branch behind one reusable public "mark sector with alternatives" API that accepts sector id/index/object and chains `onComplete`. Keep a private/internal atomic `markOnSector()` only for the final sector branch. Then migrate all non-test production callers that represent "mark a signal in a sector" to the alternative-aware API.

## Warnings

### W-01: Oumuamua tile marking bypasses normal signal bookkeeping

**File:** `packages/server/src/engine/alien/plugins/OumuamuaAlienPlugin.ts`
**Line(s):** 183-207, 392-424
**Dimension:** Logic Correctness / Code Quality

`markTileSignal()` appends an alien-board occupant and resolves completion, but it does not deploy a `SECTOR_MARKER`, return markers on reset, or record a `SIGNAL_PLACED` mission event. If the Oumuamua tile marker is intended to be a signal marker "like marking other planets/sectors", this path can ignore piece limits and miss mission/trigger integrations. The completion logic correctly removes all Oumuamua tile markers, but piece accounting should mirror that removal if pieces are deployed.

**Suggested fix:** Decide whether Oumuamua tile markers use the same sector marker inventory. If yes, deploy on `markTileSignal()`, return all deployed markers in `resolveTileCompletion()`, and record a signal event with a distinguishable target if needed.

### W-02: Planet/sector position lookup is still duplicated outside the new API

**Files:** `packages/server/src/engine/alien/plugins/AnomaliesAlienPlugin.ts`, `packages/server/src/engine/alien/plugins/AnomaliesResolver.ts`, `packages/server/src/engine/cards/base/AnySignalQuickMissionCard.ts`, `packages/server/src/engine/cards/base/HerschelSpaceObservatoryCard.ts`
**Line(s):** `AnomaliesAlienPlugin.ts` 140-156, `AnomaliesResolver.ts` 24-39, `AnySignalQuickMissionCard.ts` 178-190, `HerschelSpaceObservatoryCard.ts` 135-143
**Dimension:** Maintainability

The codebase now has `SolarSystem.getSectorIndexOfPlanet()` and `ScanEffectUtils.getSectorIndexByPlanet()`, but several modules still recalculate `Math.floor(indexInRing / ringIndex)` or manually scan spaces. This is exactly the kind of duplicated board-position logic that can drift when the coordinate model changes.

**Suggested fix:** Move generic queries to a shared server/common rules helper, for example `getSectorIndexBySpace(space)`, `getSectorByPlanet(game, planet)`, and `getSectorIdsWithPlayerProbes(game, playerId)`. Then replace the ad hoc calculations in card and alien plugins.
