# Diff Review - 2026-05-01

Scope: current uncommitted changes for Mascamites alien cards and related mission/scoring logic.

## Findings

### Warning: ET.5 pickup-back ignores Mascamites capsules even though capsules count as probes

- Status: fixed in current working tree.
- File: `packages/server/src/engine/cards/alien/MascamitesCardUtils.ts:58`
- Impact: ET.5 can be incorrectly unavailable when the player has a Mascamites sample capsule on Jupiter/Saturn but no normal probe there.
- Details: `createPickupBackInput(..., { requireOwnProbe: true })` checks only `game.solarSystem.getProbesAt(...)`. The Mascamites rules state that a sample capsule counts as a probe for card/tech effects (`docs/arch/aliens/mascamites/rule.md:36-40`), and the engine already models capsules as movable pieces (`packages/server/src/engine/freeActions/movementTargets.ts:157`). ET.5 text is "planet with your probe", so the eligibility check should include owned capsules at that planet space as well as normal probes.
- Suggested fix: extend `hasOwnProbeAtPlanet` to also check owned Mascamites capsules whose `spaceId` is on the queried planet.
- Suggested test: create a capsule on Jupiter/Saturn, remove the normal probe, play ET.5, and assert the pickup-back choice is still offered.

### Warning: Customized mission reward handling does not continue the reward chain after pickup-back

- Status: fixed in current working tree.
- File: `packages/server/src/engine/missions/MissionReward.ts:35`
- Impact: future or changed mission data with `desc.et-pickup-back-reward` followed by additional rewards, or with an `onComplete` continuation, will silently skip those later steps after the player selects a sample.
- Details: the customized reward branch returns `createPickupBackInput(...)` directly. The sample option's `onSelect` in `MascamitesCardUtils.ts:97` resolves token rewards and then returns `undefined`, so `applyAt(index + 1)` and `onComplete` are never called. Current ET.8/ET.9/ET.10 data only has this one customized reward, so the shipped card data does not expose the bug today, but the mission reward executor contract is sequential and this branch breaks that contract.
- Suggested fix: add an `onComplete` callback option to `createPickupBackInput` and call it after token reward resolution, or let `MissionReward` wrap the returned input so sample selection continues with `applyAt(index + 1)`.
- Suggested test: add a focused unit test with a pickup-back custom reward followed by a base reward, and assert both rewards resolve.
