### C-01: Centaurians milestone reward market is not implemented at all

**File:** `packages/server/src/engine/alien/plugins/CentauriansAlienPlugin.ts`, `packages/server/src/engine/alien/AlienBoard.ts`
**Line(s):** `CentauriansAlienPlugin.ts:71-97`, `AlienBoard.ts:575-608`
**Dimension:** Logic

The rules require every Centaurians message milestone to claim one reward space on the species board, including data-cost spaces and permanently blocked rewards. That state and interaction loop do not exist in the implementation.

Rule reference:
- [docs/arch/aliens/centaurians.md](/Users/oushuohuang/Documents/seti/docs/arch/aliens/centaurians.md:33)
- [docs/arch/aliens/centaurians.md](/Users/oushuohuang/Documents/seti/docs/arch/aliens/centaurians.md:74)

Current behavior:
- `CentauriansAlienBoard` stores only `messageMilestones` and `pendingMessagesByPlayer`
- `onMilestoneCheck()` resolves a delayed card effect directly
- discovery milestones with `sourceCardId: null` resolve to nothing at all

That means the implementation skips the shared reward market entirely, so the main Centaurians board mechanic is absent.

**Suggested fix:**
- Add Centaurians reward-slot state to `AlienBoard`
- model occupancy/locking and any per-space costs in `common`
- change `onMilestoneCheck()` to prompt the player to choose an available reward, pay any required data from the data pool, and only then resolve the green/delayed effect flow
- extend serializer + public protocol + client board rendering for that reward market

### C-02: Trace-style Centaurian cards disappear instead of being discarded after delayed resolution

**File:** `packages/server/src/engine/actions/PlayCard.ts`, `packages/server/src/engine/alien/plugins/CentauriansAlienPlugin.ts`
**Line(s):** `PlayCard.ts:134-141`, `CentauriansAlienPlugin.ts:119-135`
**Dimension:** Logic

Cards like `ET.34` / `ET.35` / `ET.36` / `ET.37` are removed from hand and marked as `destination: 'alien'`, but when their delayed milestone resolves, the code only creates a trace input. It never tucks them and never discards them.

Rule reference:
- [docs/arch/aliens/centaurians.md](/Users/oushuohuang/Documents/seti/docs/arch/aliens/centaurians.md:27)

So these cards effectively vanish from all zones after resolution, which is a state corruption bug.

**Current:**
```ts
if (delayed?.type === 'trace') {
  return game.alienState.createTraceInput(player, game, delayed.trace, {
    onComplete,
  });
}
```

**Suggested fix:**
- after the delayed trace resolves, discard non-income Centaurian cards to the Centaurians discard pile
- keep income-style cards on the current tuck path
- add a regression test asserting that `ET.34` ends up in the alien discard after its delayed trace is resolved

### C-03: Exertians +40 milestone spends the 1 credit before the player chooses whether to play

**File:** `packages/server/src/engine/alien/plugins/ExertiansAlienPlugin.ts`
**Line(s):** `ExertiansAlienPlugin.ts:343-369`
**Dimension:** Logic

The second Exertians milestone currently charges the 1 credit before showing the option list, and the option list still contains `skip-exertian-facedown`. If the player skips, the credit is still gone.

Rule reference:
- [docs/arch/aliens/exertians.md](/Users/oushuohuang/Documents/seti/docs/arch/aliens/exertians.md:30)

This produces an invalid state transition: a player can legally decline the optional play, but still lose the credit.

**Current:**
```ts
if (creditCost > 0) {
  if (!player.resources.canAfford({ credits: creditCost })) {
    return onComplete();
  }
  player.resources.spend({ credits: creditCost });
}
```

**Suggested fix:**
- defer the spend into the `play-facedown:*` branch
- keep the current “opportunity is lost if unaffordable” behavior when `canAfford()` is false
- add a regression test for “player has 1 credit, chooses skip, credit remains 1”

## Warnings

### W-01: Final scoring breakdown folds Exertians penalties into `alienBonus`

**File:** `packages/server/src/engine/scoring/FinalScoring.ts`
**Line(s):** `FinalScoring.ts:78-89`
**Dimension:** Quality

The new penalty path is numerically fine, but the breakdown mutates `alienBonus` by adding a negative penalty into it:

```ts
playerBreakdown.alienBonus += penalty;
```

That makes the result misleading for any UI/logging that expects `alienBonus` to mean positive alien scoring only. It also makes future debugging of endgame totals harder.

**Suggested fix:**
- add a dedicated `alienPenalty` field to `IPlayerFinalScoreBreakdown`
- leave `alienBonus` as positive-only scoring
- update serialization/UI consumers accordingly
