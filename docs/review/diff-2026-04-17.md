### Warnings

**W1 — Checkpoint lifecycle has a leak path via free actions when pipelines enter resolution.**
`Game.ts` now opens a mission checkpoint per public entry point
(`processMainAction` / `processFreeAction` / `processInput`) and closes it only when the phase
settles back to `AWAIT_MAIN_ACTION` / `GAME_OVER`. `processInput` calls
`closeMissionCheckpointIfSettled()` but has **no `try/catch` that calls `endCheckpoint()` on failure**
(compare `processMainAction`, which does). If an exception bubbles out of `runResolutionPipeline`
during `processInput`, the checkpoint stays open; subsequent `recordEvent` calls from the *next*
action will piggy-back onto the stale `checkpointId` because `ensureMissionCheckpoint` keeps the
existing id. In practice no current test exercises this path, so tests stay green, but this is a
latent bug worth hardening:

```300:315:packages/server/src/engine/Game.ts
  public processInput(
    playerId: string,
    payload: IPlayerInputPayload,
  ): IPlayerInput | undefined {
    ...
    player.waitingFor = undefined;
    this.runResolutionPipeline();
    this.closeMissionCheckpointIfSettled();
  }
```

Recommend wrapping the body in the same `try { ... } catch { endCheckpoint(); throw; }` as
`processMainAction`.

**W2 — `MissionTracker.test.ts` parses branch index out of option ids by string slicing.**
The new integration test derives the branch number from the last character of an id:

```
state116?.branchStates[
  Number.parseInt(expectedControlCenterOptionId.slice(-1), 10)
]?.completed
```

This works only because current ids end in `0..9`; any future `complete-<card>-10` (or renamed
scheme) will silently assert the wrong branch. Extract a small helper (e.g.
`parseCompleteBranchIndex(id)`) that pulls the suffix after the last `-`, or look up the branch
directly from the prompt option's `meta` if available.

**W3 — `Scan.test.ts` 2.4.10 monkey-patches `missionTracker.recordEvent`.**
The capture pattern reassigns the tracker's method:

```
const originalRecord = game.missionTracker.recordEvent.bind(game.missionTracker);
game.missionTracker.recordEvent = (event) => { captured.push(...); originalRecord(event); };
```

Per the TDD-plan anti-pattern table, this tests a mock surface rather than observable behaviour and
mutates a production singleton. There is already a concrete `SIGNAL_PLACED` regression in
`MarkSectorSignalEffect.test.ts` (as the plan's 2.4.10 coverage note points out), so either
drop this case or assert via `game.eventLog`/a test-only spy that is installed before the action and
restored in `afterEach`. The rest of `Scan.test.ts` uses public state assertions correctly; this is
the odd one out.

**W4 — `ObservationQuickMissionCard.test.ts` "fallback" path regressed in scope.**
Old test drove `card.play({ player, game })` with a stub sector and verified the card still emitted
two color-based signals when `solarSystemSetup` was null. The new test replaces that with:

```
const result = MarkSectorSignalEffect.markByStarName(player, game, EStarName.SIXTY_ONE_VIRGINIS);
expect(result).toBeNull();
```

That asserts the *primary effect class* returns `null` when the star cannot be located — a weaker
contract than the previous "card still places signals via fallback". Either (a) rename the test to
reflect the narrower contract, or (b) keep the integration flavour by playing card `37` through
`processMainAction` against a `Game` whose `solarSystemSetup` has been stripped, and asserting the
color fallback still places two markers. Today's test name ("falls back to color-based placement")
no longer matches what it checks.

**W5 — `PlayCard.test.ts` removed `canExecute`/`execute` micro-tests; the plan's own note
(2.6E.1) says those edges live in `GameIntegration.test.ts`, which is fine — but the `discard` spy
case is now nowhere:**
Legacy test `'discards the played card via mainDeck.discard'` is gone. The new integration cases
assert ordinary cards land in the discard pile (`2.6.1`), but there is no direct regression for the
internal `mainDeck.discard(cardId)` call semantics. Acceptable because that is an implementation
detail, but worth noting in case a refactor swaps discard mechanics.

### Suggestions

**S1 — Consider deleting the legacy `describe.skip` in `BehaviorExecutor.test.ts` now.**
Stream-B's own review recorded parity sign-off ("All previously mocked invariants from the legacy
block are re-asserted in the integration suite"). Per TDD-plan execution rules
("遵循 TDD Skill Iron Law: 'Don't keep it as reference. Delete means delete.'"), the two skipped
cases should go. Leaving `.skip` indefinitely is the exact failure mode the plan warns about.

**S2 — `getMissionTriggerPlayers` returns a fresh array every call via `.map`;** acceptable but a
2-player game hits this per main action. Not a perf concern but could be cached per checkpoint if
we later see hot-path noise.

**S3 — `ScanActionPool.ts` now hides `DONE` until `MARK_EARTH` is executed.** This enforces the
rule (FAQ "marking the earth sector is mandatory") and is covered by 2.4.8 tests. One UX caveat:
when the card row is empty and the player has no tech that provides extra marks, the menu
degenerates to a single-option `SelectOption`. Clients must handle "menu with one option that
auto-resolves" gracefully — worth a quick check in the client's action-menu renderer before the
next E2E run.

**S4 — `ScanEffectUtils.extractSectorColorFromCardItem` now accepts `string` card ids via
`loadCardData`.** The helper loads card data on every call; if `cardRow` ever grows or the
function is called in a tight loop (e.g. tech-driven multi-scan), consider memoizing by cardId.
Not urgent — current call sites are per-scan.

**S5 — `MissionTracker.drainCheckpoints` relies on checkpoint ids being strictly monotonic and
contiguous in `eventBuffer`.** That holds today because `runInCheckpoint` is always called from a
single JS turn (no interleaving). A short invariant comment on `recordEvent` would save future
debugging if someone introduces async event recording.

**S6 — `client/__tests__/api/` is untracked but referenced from `docs/TODO.md`.** Make sure those
regressions are committed in the same PR so the TODO entry's "Regression coverage" link is not
dangling.
