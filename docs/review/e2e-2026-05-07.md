## Warnings

### W-01: Auth API tests depend on previous tests creating shared state

**File:** `packages/e2e/tests/auth.spec.ts`
**Line(s):** 15, 23-45, 69-79
**Dimension:** Quality / Accuracy

`uniqueEmail` is created once for the whole `describe`, then later tests assume the first registration test has already created that user. For example, `login with registered user via API`, `reject duplicate registration`, and `GET /auth/me succeeds with valid token` all reuse the same email. This makes the tests order-dependent: running one test alone, retrying after partial failure, or parallelizing the file can produce false failures or false positives.

**Suggested fix:** Make each test create its own prerequisite user inside that test or in a per-test fixture. For duplicate registration, register once inside the test, then assert the second registration returns `409`. For login/profile, register the user in the same test before logging in.

---

### W-02: Responsive and reconnect acceptance criteria are currently missing

**File:** `packages/e2e/playwright.config.ts`
**Line(s):** 25-29
**Dimension:** Completion / Coverage

Stage 9-4 explicitly asks for desktop/tablet/mobile responsive checks and one reconnect case. The current Playwright project list only contains Desktop Chrome. A search across `packages/e2e/tests` shows no `setViewportSize`, `mobile`, `tablet`, or reconnect test. `WsTestClient` even disables Socket.IO reconnection, and no spec imports it.

**Suggested fix:** Add either named Playwright projects for `desktop`, `tablet`, `mobile`, or a focused responsive spec that sets the three viewports. Add one browser-level reconnect spec that starts from a real game page, disconnects/reloads or interrupts the socket, and asserts state recovery from the production path.

---

### W-03: Multiple smoke/journey specs repeat the same `register -> room -> game -> PASS` path

**Files:** `packages/e2e/tests/game-flow.spec.ts`, `packages/e2e/tests/game-flow-user-path.spec.ts`, `packages/e2e/tests/game-flow-behavior.spec.ts`, `packages/e2e/tests/browser-smoke.spec.ts`, `packages/e2e/tests/user-journey.spec.ts`, `packages/e2e/tests/fixed-user-smoke.spec.ts`, `packages/e2e/tests/game-actions.spec.ts`
**Dimension:** Maintainability / Redundancy

These files repeatedly create two browser contexts, register host/guest, create a room, join, launch, enter game, then assert dashboard visibility and/or first PASS sync. Some add screenshots or diagnostics, but the behavioral contract is effectively the same. This makes failures noisy and increases runtime without proportionally increasing confidence.

**Suggested fix:** Keep one canonical production-path smoke with diagnostics, one focused PASS sync test, and one multi-player room/rotation test. Fold `game-flow`, `game-flow-user-path`, and `game-flow-behavior` into a single named core journey unless each gets a unique assertion beyond setup and PASS.

---

### W-04: Seed/scenario room creation is duplicated via local route body rewriting

**Files:** `packages/e2e/tests/main-action-analyze-data.spec.ts`, `packages/e2e/tests/alien-discovery-real-flow.spec.ts`, `packages/e2e/tests/free-action-complete-mission.spec.ts`
**Dimension:** Maintainability / Accuracy

Several specs defined their own `createRoomByUiWithSeed` / `createRoomByUiWithScenario`, each opening the create room dialog, intercepting `**/lobby/rooms`, mutating the POST body, submitting, then un-routing. This was hard to audit and easy to let drift from the actual UI helper. It also hid that scenario-preset tests depended on a non-public setup shortcut.

**Suggested fix:** Extend `createRoomByUiWithDetails` to accept explicit `seed` test options and centralize route mutation in one helper with `try/finally` cleanup. Do not support public `scenarioPreset` in browser E2E; server-side tests explicitly verify public room creation ignores it.

---

### W-05: `real-flow.ts` mixes too many responsibilities and duplicates prompt resolution

**File:** `packages/e2e/helpers/real-flow.ts`
**Line(s):** 49-140, 181-393, 480-724
**Dimension:** Maintainability

`real-flow.ts` is now a 724-line helper containing user creation, auth UI, room creation, alien pool selection, room join/launch/enter retries, action ownership polling, event log checks, main action clicks, end turn, generic prompt clicking, and scan sub-action resolution. It also has two very similar card prompt resolvers. This makes tests shorter, but it concentrates too many hidden behaviors in one file and makes "accurate but not redundant" hard to evaluate.

**Suggested fix:** Split by workflow boundary: `auth-flow.ts`, `room-flow.ts`, `game-session.ts`, `prompt-resolvers.ts`, and `assertions.ts`. Merge `resolveCardPromptIfVisible` and `waitForAndResolveCardPrompt` into one API with explicit behavior, and keep scan-specific heuristics in a named scan helper.

---

### W-06: Database setup is duplicated across Playwright config, global setup, and local scripts

**Files:** `packages/e2e/playwright.config.ts`, `packages/e2e/global-setup.ts`, `scripts/run-e2e-local.sh`, `scripts/init-e2e-local.sh`
**Dimension:** Maintainability / Reliability

Local Playwright config starts the server with `db:prepare:e2e`, then `globalSetup` also runs `db:prepare:e2e`. The local wrapper script separately calls `init-e2e-local.sh`, which also prepares the DB before starting services. This is redundant and makes it unclear which command is authoritative for DB reset semantics.

**Suggested fix:** Choose one owner for DB preparation per entry point. For example, keep DB prep in `init-e2e-local.sh` for wrapper runs and use `SKIP_E2E_DB_PREPARE=1` when the wrapper invokes Playwright, or keep it in Playwright `globalSetup` and remove it from the server command.

## Suggestions

### S-01: Separate production-path helpers from debug/API shortcuts

**Files:** `packages/e2e/helpers/api.ts`, `packages/e2e/helpers/auth.ts`, `packages/e2e/helpers/ws-client.ts`
**Dimension:** Maintainability

`api.ts` contains production auth/lobby methods and many debug endpoints. `auth.ts` exposes localStorage injection, and `ws-client.ts` exposes raw socket actions, but current specs do not import those shortcut helpers. Keeping them beside production-flow helpers makes it harder to enforce the project's "real UI path" rule.

**Suggested fix:** Move debug APIs to `helpers/debug-api.ts`, localStorage auth injection to `helpers/test-only-auth.ts`, and raw socket helpers to `helpers/ws-shortcuts.ts`, or delete unused helpers until a specific test needs them.

---

### S-02: Define a small E2E suite taxonomy

**File:** `packages/e2e/tests`
**Dimension:** Maintainability

The suite currently mixes API tests, production UI journeys, deterministic seeded UI tests, scenario-preset UI tests, debug replay tests, and smoke diagnostics in one flat folder. The filenames are mostly descriptive, but the intended run groups are not obvious.

**Suggested fix:** Add naming or folders such as `smoke/`, `flows/`, `actions/`, `debug/`, and `api/`, or add Playwright tags like `@smoke`, `@real-ui`, `@debug`, `@slow`. This lets CI run a minimal smoke set while preserving broader coverage for scheduled/full runs.

---

### S-03: Prefer one canonical setup fixture over per-file setup factories

**Files:** `packages/e2e/tests/*`
**Dimension:** Maintainability

Several files define local variants of `openUserContext`, `setupTwoPlayerGame`, or `createStartedTwoPlayerGame`. A small fixture that returns `{ host, guest, pages, roomId, gameId }` would reduce boilerplate while still allowing tests to opt into seed/scenario/alien pool options.

**Suggested fix:** Add a typed `createStartedGameByUi(browser, options)` fixture/helper and let specs declare only the part that is unique to the action being tested.

## Recommended Consolidation

Keep as high-value core coverage:

- `browser-smoke.spec.ts` or one renamed equivalent as the diagnostic smoke.
- `alien-discovery-real-flow.spec.ts` for the strongest real alien UI/rules closure.
- Focused main/free action specs that assert real state changes, especially analyze data, orbit/land, research tech, complete mission, movement, exchange, buy card, card corner, and place data.
- `round-rotation.spec.ts`, `end-of-round-cycle.spec.ts`, and `room-multiplayer.spec.ts` for multiplayer/turn semantics.
- `debug-replay.spec.ts` and `debug-snapshot-replay.spec.ts` as explicitly debug-only support coverage.

Consider merging or deleting after consolidation:

- `game-flow.spec.ts`, `game-flow-user-path.spec.ts`, `game-flow-behavior.spec.ts`, `fixed-user-smoke.spec.ts`, and the overlapping PASS case in `game-actions.spec.ts`, unless each is given a distinct assertion that is not already covered by the canonical smoke.
